self hosting

November 7, 2025 · 3 min

If you need a machine and don’t buy it, then you will ultimately find that you have paid for it and don’t have it.
— Henry Ford via Clayton Christensen

These days I’m working on an HTTP server in C using WebSockets, just to get a taste of how things work under the hood.
The only functionality of the server is to listen and send HTML and media files.

Now a days cloud is more popularized for its accessibility, scalability, and the fact that real hardware is abstracted away from developers.
I wanted to see for myself what it’s like to self-host on a bare-metal server — even though it’s a small project.
At this point, my servers are sitting idle anyway, so why not?

Note#

If you are planning to self-host your personal projects, the steps are pretty much the same — all you need is a host.

FOR Front-End Developers#

You probably don’t want to manage your own server. In that case, you can go with serverless platforms, which handle deployment automatically.

These are the easiest options for frontend-focused developers. Popular options:

  • Vercel
  • Netlify
  • AWS Amplify

Though most of these services are basically just wrappers around AWS, offering a nicer UI and developer experience while charging more for the convenience.

The thing is, I had no idea where to begin, so I researched for a couple of days about how to self-host.
Here’s what I concluded:

Typically, the process involves port forwarding and requires a static IP.
Guess what? Most telecom services don’t provide static IPv4 for home consumers since they’re behind CGNAT.
Even I didn’t know what that actually mean.

Then I came to know about web tunnels, where you can forward all incoming requests to a server running on your localhost.

NOTE:Tunnels are okay for deploying demo not recommended for the production in case you have consider amount of traffic on your site

Everything worked fine, and now I just had to create a process that always runs in the background on boot — without any user intervention.
This is what’s called a daemon process in Linux.

“A daemon is a background process that handles requests for services like file transfers, printing, or managing network connections — kind of like a waiter at a restaurant, serving requests without you ever noticing.
The word itself comes from the ancient Greek meaning an attendant spirit.
Invisible, but always there.
My daemon — it’s always there too. Watching. Guiding. Sometimes I think about how much like them I am… always running in the background.”
— Elliot, Mr. Robot

Creating your own daemon#

So I SSH’d into my remote machine (btw, SSH also has its own daemon process called sshd, which always listens for incoming connections).
Here’s what I learned about making a daemon service of your own:

Create a new service file:


   sudo vim /etc/systemd/system/<name>.service

Configure it with your executable path, working directory, and restart policy. from this point managing the service process are same as any other daemon process

1.sudo systemctl start <service name>      
2.sudo systemctl status <service name>    
3.sudo systemctl enable <service name>

if you got any problem activating service.
try

sudo journalctl -xe

which is also a daemon process which keeps logs of the other daemons.
